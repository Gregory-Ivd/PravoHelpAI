## 2026-05-03 — Базові рішення Фази 0

### D-001. Тип проєкту: реальний запуск, не навчальний.
**Рішення:** проєкт буде використовуватись реальними користувачами.
**Наслідки:**
- Шаблони документів **обовʼязково** проходять ревʼю юриста (Дмитро Глушко) перед публічним релізом.
- Юридичний дисклеймер у `/start` — обовʼязковий, сильний, з кнопкою-підтвердженням.
- Персональні дані (ПІБ, ІПН, зарплата, паспорт) обробляємо обережно: видаляємо тимчасові DOCX через 1 год, шифрування БД у v1.1.
- Хостинг: локально тільки на фазі розробки. Для публічного запуску — VPS.

### D-002. Сценарії MVP — фінальний список.
**Рішення:** 3 сценарії:
1. **Невиплата зарплати** (трудове право — стабільне).
2. **Оскарження штрафу за порушення ПДР** (адмінправо — стабільне).
3. **Повістка / мобілізаційні питання** (військове право — змінюється; з посиленим дисклеймером).

**Що змінилось:** початково я (Claude) пропонував замінити «повістку» на «повернення коштів за товар». Користувач наполіг залишити повістку через її актуальність. Компроміс: лишаємо повістку, але:
- Кожен шаблон містить поле `актуально_станом_на: [дата]`.
- Окремий рядок у дисклеймері: «Військове законодавство України оновлюється часто. Перевірте актуальність на rada.gov.ua перед поданням.»
- В кінці сценарію «повістка» **підсвічена** кнопка «Зв'язатись з Дмитром Глушком» — консультація юриста рекомендована як необхідний крок.

### D-003. Початкова форма: ConversationHandler у боті, не Google Form.
**Причина:** UX, валідація на льоту, дані одразу в state, без виходу з Telegram.

### D-004. AI/LLM на старті: НІ.
**Причина:** галюцинації статей закону неприпустимі для юридичного документа. MVP — чисто детерміновані шаблони через `docxtpl`. LLM додаємо в v1.1 (опціонально, для перефразування пояснень — НЕ для генерації самого юридичного тексту).

### D-005. Формат документів: DOCX тільки в MVP.
**Причина:** юрист може редагувати шаблон, користувач — допрацьовувати. PDF додамо за потреби в v1.1 (через LibreOffice CLI або docx2pdf).

### D-006. Передача документів: НЕ автоматична.
**Рішення:** бот віддає DOCX + чек-лист «куди і як подати». Жодних email/SMS від бота.
**Причина:** юр. ризики, спам-фільтри, відсутність ЕЦП-інтеграції в MVP.

### D-007. Юрист-партнер: Дмитро Глушко.
**Канал контакту:** телефон + Telegram (конкретні дані додамо в `.env` коли отримаємо).
**Спеціалізація:** користувач уточнить. До уточнення — припускаємо, що Дмитро покриває всі 3 сценарії.
**Згода:** Дмитро погодився приймати лідів з бота.
**Шаблони:** Дмитро надасть приклади документів — користувач передасть йому запит (див. `MESSAGE_FOR_DMITRO.md`).
**Інтеграція:** наприкінці кожного сценарію — кнопка «Зв'язатись з Дмитром». Для сценарію «повістка» кнопка підсвічена і рекомендована.

### D-008. Технічний стек.
- Python 3.11+
- `python-telegram-bot` v21+ (async, polling в dev, webhook в prod)
- SQLite + SQLAlchemy 2.x
- `docxtpl` для DOCX
- `python-dotenv` для секретів
- `structlog` для логів
- `pytest` + `pytest-asyncio` для тестів
- Без Redis, Celery, Docker, LLM (поки)

### D-009. Хостинг — план двофазний.
**Фаза розробки:** локально на ПК користувача (polling).
**Фаза публічного запуску:** VPS (Hetzner CX11 ~5 €/міс або аналог), polling → webhook, systemd-юніт.
**Чому не безкоштовний хостинг (Railway/Fly.io free tier):** засинання інстансу, обмеження на network, можуть забанити за «юридичну консультацію» при скаргах. VPS дешевий і повністю під контролем.

### D-010. Git і репозиторій.
**Рішення:** приватний GitHub-репозиторій. Локальний `git init` стартуємо одразу, GitHub-remote підключаємо після створення репо.
**Власник репо:** Gregory (owerhint@gmail.com).
**Майбутні співавтори:** Дмитро Глушко як reviewer на ревʼю шаблонів (опціонально, після його згоди).

### D-011. Термін MVP — 2 тижні з фокусом на 1 сценарій.
**Бюджет:** 4 год/день × 14 днів = ~56 годин.
**Реалістичний обсяг:**
- Дні 1–2: Фаза 0 + Фаза 1 (скелет, БД, /start, дисклеймер) — ~7 год.
- Дні 3–8: Фаза 2 «невиплата зарплати» — ~24 год. Робочий MVP з одним сценарієм до кінця тижня 2.
- Дні 9–14: Фаза 2.5 «повістка» — **умовно**, тільки якщо Дмитро надіслав шаблони і провів ревʼю. Інакше — виносимо в наступний цикл.
- Сценарій «штраф ПДР» — поза обсягом цих 2 тижнів.

### D-012. Перший сценарій MVP — невиплата зарплати.
**Рішення:** перший і базовий сценарій — невиплата зарплати.
**Чому не повістка (хоч користувач спочатку запропонував):**
- Стабільне трудове законодавство (КЗпП ст. 116, 117) проти найбільш мінливого військового.
- Шаблони доступні з офіційних джерел (Держпраці, ЄРСР) — не блокуємось на Дмитрові.
- Низький ризик для користувача навіть при «середній» якості шаблону.
- Доводимо архітектуру стека на простому сценарії, потім масштабуємо на складне.
**Повістка — другий сценарій,** запускаємо як тільки Дмитро надішле шаблони.

### D-013. Версія Python.
**Поточна на машині:** 3.14.3.
**План:** використовуємо системну 3.14, якщо `python-telegram-bot` встановлюється чисто. Якщо є несумісність — створюємо venv з Python 3.12.
