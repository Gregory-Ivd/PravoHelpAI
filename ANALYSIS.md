# PravoHelpAI — критичний аналіз перед стартом коду

Мета цього документу — **зупинити** написання коду, поки не приймемо ключові рішення. Текст від «команди» (з твого промпту) — це маркетинг + дизайн-thinking framework. Він **не відповідає** на технічні питання, від яких залежить, чи буде MVP реальним або симуляцією.

Нижче — те, про що команда не написала, але без чого MVP не зробити чесно.

## 1. Що насправді просить користувач (декомпозиція ТЗ)

Твоє ТЗ містить **6 функціональних вимог**, але кожна з них приховує рішення:

| # | Вимога | Приховане рішення |
|---|---|---|
| 1 | 3 тестові сценарії | **Які саме?** Команда пропонує «повістка / зарплата / штраф». Це найгірша комбінація для MVP — див. §3. |
| 2 | Початкова форма | **Google Form чи в боті?** Команда не визначилась. Це критично для UX. Див. §4. |
| 3 | Уточнюючі питання | **Дерево рішень чи LLM-діалог?** Це різна архітектура і різні ризики. Див. §5. |
| 4 | «Кілька планів вирішення» | **Що це означає?** Альтернативи (досудове / суд / скарга в орган)? Або просто чек-лист дій? Це різна складність. |
| 5 | Документ для відправки | **PDF чи DOCX?** **Один чи кілька?** **Куди надсилати — сам користувач чи бот через email?** Див. §6. |
| 6 | «Глибокий критичний аналіз» | Цей документ. ✅ |

**Висновок:** ТЗ виглядає простим, але насправді в ньому ~12 архітектурних рішень. Без відповідей — MVP буде або переусклажненим, або несправжнім.

## 2. Що НЕ є частиною MVP (явно виключаємо)

З тексту команди багато про **маркетинг, PR, growth, аналітику CAC/LTV, A/B тести, TikTok-контент**. Це все **не стосується технічного MVP** і не входить у скоп цього проєкту. Виносимо в out-of-scope, щоб не розмивати фокус:

- ❌ Платежі / freemium / підписки
- ❌ Веб-сайт, SEO, лендинг
- ❌ Мобільний додаток
- ❌ Особистий кабінет / auth
- ❌ Маркетингові інтеграції (UTM, аналітика трафіку)
- ❌ Multi-language (тільки українська)
- ❌ Real-time чат з юристом
- ❌ Push-нагадування про строки (хороша ідея, але v2)

**Це нормально.** MVP — це **мінімум, що дає реальну цінність одній людині**. Все інше — пізніше.

## 3. Критика вибору 3 сценаріїв

Команда пропонує: **повістка / невиплата зарплати / штраф**. Це **погана комбінація** для MVP. Розбираю по кожному:

### 🔴 Сценарій «Повістка» — НАЙРИЗИКОВАНІШИЙ
- Військове законодавство України **змінюється квартально** (мобілізаційний закон 2024, реєстр «Оберіг», ЕЦК через «Резерв+», штрафи 17–25 тис. ₴, СЗЧ).
- Помилкова порада тут = **кримінальна відповідальність** для користувача.
- Тут ніхто, крім адвоката з військовою спеціалізацією, не дасть надійного шаблону.
- **Без юриста-консультанта в команді цей сценарій робити НЕ МОЖНА** навіть як MVP.

**Рекомендація:** замінити на щось менш політично гаряче (наприклад: «розірвання договору оренди» або «повернення коштів за неякісний товар»).

### 🟡 Сценарій «Невиплата зарплати» — НАЙКРАЩИЙ ДЛЯ MVP
- Стабільна нормативка: КЗпП, ст. 116, 117.
- Чіткий шаблон документа: заява до Держпраці / позов до суду.
- Низький ризик — навіть «середній» шаблон не нашкодить користувачу.
- Це і є **«ідеальний MVP-кейс»**.

### 🟡 Сценарій «Штраф» — РОЗПЛИВЧАСТИЙ
- «Штраф» — це **категорія**, не сценарій. Бувають:
  - адмінштраф ДАІ/ТСЦ (КУпАП ст. 122)
  - штраф ТЦК (мобілізаційний — див. вище 🔴)
  - податковий штраф (ПКУ)
  - ОСББ / комуналка
- Кожен — окрема процедура оскарження. Треба **звузити** до одного типу.

**Рекомендація:** «оскарження штрафу за порушення ПДР» — найбільш поширений і зрозумілий.

### ✅ Моя пропозиція 3 сценаріїв для MVP
1. **Невиплата зарплати** — заява до Держпраці + претензія роботодавцю.
2. **Оскарження штрафу за порушення ПДР** — скарга в районний суд (ст. 287–289 КУпАП).
3. **Повернення коштів за неякісний товар** — претензія + позов до суду (ЗУ «Про захист прав споживачів»).

Це 3 сценарії з:
- стабільною нормативкою (роки без змін),
- чіткими шаблонами документів,
- низьким ризиком для користувача навіть при середній якості шаблону,
- різною механікою (трудове / адміністративне / цивільне) — добре покриває різні гілки UX.

**Запитання користувачу:** погоджуєшся на цю заміну, чи наполягаєш на оригінальних 3?

## 4. Початкова форма: Google Form vs в боті

| Критерій | Google Form | В боті (ConversationHandler) |
|---|---|---|
| UX | ❌ Користувач виходить з Telegram, повертається через лінк | ✅ Все в одному вікні |
| Швидкість запуску | ✅ 30 хв | 🟡 1–2 год коду |
| Валідація даних на льоту | ❌ Тільки наприкінці | ✅ На кожному кроці |
| Зміна питань | ✅ Через UI Google | 🟡 Правки коду |
| Інтеграція з логікою бота | ❌ Треба парсити з Sheets API | ✅ Одразу в state |
| Збереження приватності | ❌ Дані в Google | ✅ У твоєму SQLite |

**Рекомендація: в боті.** Google Form — це костиль, який ускладнить інтеграцію. ConversationHandler у `python-telegram-bot` v21 робить це елегантно.

**Виняток:** якщо ти хочеш, щоб юрист/маркетолог **бачив відповіді в таблиці без коду** — тоді Google Form. Інакше — в боті.

## 5. Дерево рішень vs LLM-діалог

Команда обіцяє «AI-сервіс на OpenAI/Claude». Це маркетинг, і він небезпечний для юридичних документів.

### Чому НЕ можна довіряти LLM генерацію юридичного тексту:
- Галюцинації статей закону (LLM придумує «ст. 158-2 КЗпП», якої немає).
- Неактуальна нормативка (відсічка знань — старі статті).
- Юридичний текст вимагає **точних реквізитів**: суд, адреса, дата, ПІБ, посилання — LLM псує форматування.
- Користувач не зможе перевірити, чи правда у згенерованому документі.

### Як правильно для MVP:
1. **Дерево рішень (FSM)** для збору даних — детерміноване, передбачуване.
2. **Шаблон документа** (DOCX з плейсхолдерами `{{ім'я}}`, `{{сума}}`) — заповнюється детерміновано через `docxtpl`.
3. **LLM (опціонально, на v1.1)** — тільки для:
   - перефразування пояснень простою мовою,
   - класифікації відкритих відповідей користувача (наприклад: «опиши ситуацію» → витягнути сутності),
   - **НЕ** для генерації самого юридичного тексту.

**Рекомендація MVP v1.0:** **взагалі без LLM.** Чисті шаблони + дерево питань. Це швидше, дешевше, безпечніше. LLM додаємо в v1.1, якщо буде потрібно.

**Запитання користувачу:** ти OK з MVP без AI на старті, чи хочеш одразу інтегрувати Claude/OpenAI? (Я наполягаю на «без AI на старті».)

## 6. Формат документа: PDF vs DOCX, один чи кілька

### PDF
- ✅ Готовий до друку
- ❌ Неможливо редагувати (а користувач захоче дописати свої деталі)
- ❌ Складніше генерувати з шаблону (reportlab — програмний, шаблон — окрема історія)

### DOCX
- ✅ Користувач може дописати/виправити
- ✅ Юрист може редагувати шаблон без коду
- ✅ `docxtpl` — простий і потужний (Jinja2 в DOCX)
- ✅ Конвертація в PDF легка (через LibreOffice/Word, якщо треба)
- ❌ Word потрібен користувачу (але всі мають)

**Рекомендація: DOCX через `docxtpl`.** Можна додати кнопку «отримати ще й PDF» — конвертація через `docx2pdf` (потребує MS Word на сервері) або LibreOffice CLI.

### Скільки документів за один сценарій?

Для кейсу «зарплата» це може бути:
1. **Претензія роботодавцю** (досудовий етап).
2. **Заява до Держпраці** (контролюючий орган).
3. **Позовна заява до суду** (якщо претензія не спрацювала).

Тобто **3 документи в одному сценарії**. Це і є те, що ти називаєш «кілька планів вирішення» — користувач обирає один план (або всі), бот генерує відповідні документи.

## 7. Куди користувач надсилає документ

Команда пропонує «надсилання на адресу призначення» — **найгірша ідея для MVP**.

### Чому НЕ робимо автоматичну відправку:
- Email від бота → потрапить у спам.
- Бот без юридичної особи не може **офіційно** відправляти документи від імені користувача.
- Підпис ЕЦП — окрема велика історія (Дія.Підпис, токен, потрібен сертифікат).
- Ризик: користувач думає, що бот «відправив», а лист не дійшов.

### Як правильно для MVP:
- Бот віддає DOCX (+ опціонально PDF).
- Бот пише **інструкцію**: «1. Роздрукуй. 2. Підпиши. 3. Відправ Укрпоштою з описом вкладення на адресу `[адреса]`. 4. Зберігай чек.»
- Або: «Електронний варіант: 1. Підпиши через Дію.Підпис. 2. Надішли через `[посилання на електронний кабінет органу]`.»

**Рекомендація:** тільки інструкція + готовий документ. Жодних email від бота.

## 7.bis. Передача до реального юриста (Дмитро Глушко)

Це **дуже важлива фіча**, яку команда впустила. Після генерації документа бот пропонує:

> 💼 **Хочеш, щоб документ і твою ситуацію перевірив реальний юрист?**
> Можемо звʼязати з **Дмитром Глушком** — практикуючий юрист, спеціалізація: [...].
> [Кнопка: Зв'язатись] [Кнопка: Ні, дякую]

### Чому це сильно покращує продукт:
- ✅ **Знімає юридичну відповідальність** з бота: бот = «генератор шаблону», складні питання → юрист.
- ✅ **Reality check** для користувача: «бот зробив 80%, але є нюанси — поговори з людиною».
- ✅ **Шлях до монетизації** (на майбутнє): % з угоди / фікс за лід.
- ✅ **Канал перевірки якості шаблонів**: юрист бачить, що генерує бот, дає фідбек.

### Технічно (просто):
- Просто кнопка з контактами: Telegram-юзер `@username`, телефон, email — що дасть Дмитро.
- Або: бот пересилає коротке резюме кейсу + згенерований документ напряму Дмитру в особисті повідомлення (через `forward` або Bot API).
- Або: лінк на Calendly / Google-форма для запису на консультацію.

### Що треба узгодити з Дмитром ПЕРЕД релізом:
1. **Згода** — він взагалі готовий приймати лідів з бота?
2. **Контакт** — який канал дає (Telegram username / телефон / email / Calendly)?
3. **Спеціалізація** — для яких сценаріїв його рекомендуємо? Якщо він не з трудового, не варто кидати йому «зарплатні» кейси.
4. **Формат передачі** — повне резюме + документ, чи тільки контакт юзера, а Дмитро сам розпитує?
5. **Платно/безоплатно для перших користувачів** — це впливає на conversion в кнопку «Зв'язатись».
6. **Дисклеймер**: бот має чітко казати «це окрема послуга юриста, не безкоштовна / умови такі-то».

### Запитання користувачу
- ✅ Дмитро вже знає про проєкт і згоден?
- ✅ Який канал звʼязку даємо?
- ✅ Яка його спеціалізація — щоб зрозуміти, в яких сценаріях рекомендувати?

Якщо Дмитро **поки тільки потенційний партнер** — все одно реалізуємо кнопку, але з заглушкою «незабаром буде доступно» в MVP, і активуємо коли отримаємо згоду.

## 8. Технічний стек для MVP

### Бекенд
- **Python 3.11+** — стабільно, є вся бібліотечна інфраструктура.
- **`python-telegram-bot` v21+** — найзрілий бібліотечний вибір, async-first.
  - `ConversationHandler` для FSM (стани діалогу).
  - `JobQueue` для нагадувань (v1.1).
- **SQLite** — для MVP більше не треба. PostgreSQL — коли користувачів буде >1000.
- **`SQLAlchemy 2.x`** — ORM. Альтернатива: чистий `sqlite3`, але SQLAlchemy дає міграції через Alembic, як виростемо.
- **`docxtpl`** — генерація DOCX з шаблонів (Jinja2 синтаксис у DOCX).
- **`python-dotenv`** — `.env` для токену бота і API-ключів.
- **`structlog`** — структуровані логи в JSON.

### Без чого можемо обійтись (поки що)
- ❌ Redis (FSM зберігаємо в SQLite).
- ❌ Celery (нагадування через `JobQueue` бота).
- ❌ Docker (поки локально на твоїй машині — не треба).
- ❌ Webhook (polling достатньо для розробки і перших 100 користувачів).
- ❌ AI/LLM (див. §5).

### Тестування (важливо для тебе як QA)
- **`pytest`** — основний фреймворк.
- **`pytest-asyncio`** — для async-handlers.
- **`python-telegram-bot[test]`** — є офіційний тестовий клієнт.
- **Стратегія тестів:**
  - Unit: логіка генерації документа (на основі словника відповідей → DOCX).
  - Integration: повний прогон сценарію через mock-Update.
  - E2E (вручну): запуск бота локально + перевірка в Telegram.

|---|---|---|---|
| Шаблони документів некоректні юридично | 🔴 Висока (без юриста) | 🔴 Серйозна (шкода юзеру) | Дисклеймер у /start + НЕ публікувати без перевірки юристом |
