## Фаза 2 — Сценарій 1: невиплата зарплати (3–4 дні)

**Мета:** перший повноцінний end-to-end сценарій. Це ядро MVP — після нього 2 і 3 копіюються за шаблоном.

### 2.1. Дерево діалогу (FSM через `ConversationHandler`)
- [ ] State: `SALARY_INTRO` — пояснення, що буде зібрано.
- [ ] State: `SALARY_EMPLOYER_NAME` — назва роботодавця.
- [ ] State: `SALARY_EMPLOYER_EDRPOU` — ЄДРПОУ (опціонально, валідація 8 цифр).
- [ ] State: `SALARY_AMOUNT` — сума заборгованості (валідація: число > 0).
- [ ] State: `SALARY_PERIOD` — період заборгованості (місяць-рік від → місяць-рік до).
- [ ] State: `SALARY_LAST_PAYMENT_DATE` — дата останньої виплати.
- [ ] State: `SALARY_USER_NAME` — ПІБ користувача.
- [ ] State: `SALARY_USER_TAX_ID` — ІПН (валідація: 10 цифр).
- [ ] State: `SALARY_USER_ADDRESS` — адреса для відповіді.
- [ ] State: `SALARY_PLAN_CHOICE` — обрати з 3 планів дій (див. 2.2).
- [ ] State: `SALARY_GENERATE` — генерація документа(-ів).
- [ ] State: `SALARY_LAWYER_OFFER` — пропозиція звʼязатись з Дмитром.

### 2.2. Три плани вирішення
1. **План «М'який»**: претензія роботодавцю → 7 днів очікування → за необхідності далі.
2. **План «Контролюючий орган»**: заява до Держпраці → паралельно претензія.
3. **План «Судовий»**: позов до місцевого суду → стягнення через виконавчу службу.

Бот пояснює різницю простою мовою, користувач обирає, бот генерує **відповідні документи** + чек-лист дій.

### 2.3. Шаблони DOCX
- [ ] `templates/salary_claim_employer.docx` — претензія роботодавцю.
- [ ] `templates/salary_claim_labor_office.docx` — заява до Держпраці.
- [ ] `templates/salary_court_claim.docx` — позовна заява.
- [ ] У кожному — Jinja-плейсхолдери `{{ employer_name }}`, `{{ amount }}` тощо.
- [ ] Дисклеймер у футері кожного шаблону: «Шаблон згенеровано автоматично. Перед подачею — рекомендуємо перевірку юристом».

### 2.4. Генератор
- [ ] `document/generator.py` — функція `render(template_name, context) -> Path` через `docxtpl`.
- [ ] Файл зберігається в `data/output/{user_id}/{timestamp}.docx`.
- [ ] Бот надсилає документ як файл (`send_document`).

### 2.5. Інструкція до подачі
- [ ] Після генерації бот надсилає окреме повідомлення-чек-лист:
  - Як підписати (вручну / через Дію.Підпис).
  - Куди надіслати (адреса, реквізити).
  - Які додатки треба (копія паспорта, договір, докази заборгованості).
  - Скільки чекати на відповідь.

### 2.6. Передача до юриста
- [ ] Кнопка `Хочу консультацію Дмитра Глушка`.
- [ ] При натисканні: бот пише користувачу контакт (узгоджений у Фазі 0) + опціонально пересилає Дмитру резюме кейсу.

### 2.7. Тести
- [ ] Unit-тест: генератор створює DOCX з валідними плейсхолдерами при коректних даних.
- [ ] Unit-тест: валідація ІПН, ЄДРПОУ, суми.
- [ ] Integration-тест: повний прогон сценарію через mock-бот.

**Критерій готовності:**
- Користувач проходить весь сценарій від `/start` до отримання DOCX.
- Документи відкриваються в Word без помилок.
- Дисклеймер видно і в чаті, і в документі.
- Кнопка «юрист» працює (хоча б показує контакт).

## Фаза 3 — Сценарії 2 і 3 (3–4 дні)

**Мета:** масштабувати робочий патерн на ще 2 сценарії.

### Сценарій 2: оскарження штрафу за порушення ПДР
- [ ] FSM: дані постанови (номер, дата, орган, стаття КУпАП), обставини, докази.
- [ ] Шаблон: `fine_appeal_court.docx` — скарга в районний суд (10 днів з моменту вручення).
- [ ] Чек-лист: куди подавати, як зібрати докази (відеореєстратор, свідки).

### Сценарій 3: повернення коштів за неякісний товар
- [ ] FSM: дані продавця, дата і місце купівлі, ціна, опис недоліку, чи є чек/гарантія.
- [ ] Два шаблони:
  - `consumer_claim.docx` — претензія продавцю.
  - `consumer_court_claim.docx` — позов до суду (якщо претензія не спрацювала).
- [ ] Чек-лист: ЗУ «Про захист прав споживачів», 14 днів на повернення без причини, гарантійний строк.

### Спільне
- [ ] Кнопки «юрист Дмитро» в кінці кожного сценарію.
- [ ] Логування кожного завершеного сценарію в БД (без PII в логах).

**Критерій готовності:** усі 3 сценарії доступні з головного меню, працюють незалежно, генерують свої документи.

## Фаза 5 (опціональна) — підготовка до продакшна

> Це **ВЖЕ НЕ MVP**. Робимо тільки коли є запит від реальних користувачів.

- Хостинг: VPS (Hetzner ~5 €/міс), systemd-юніт, polling → webhook.
- Шифрування БД (SQLCipher).
- Резервні копії шаблонів і БД (cron → S3 / локально).
- Моніторинг: hands-on Sentry.
- Юрист на ревʼю шаблонів (необхідно ПЕРЕД будь-яким публічним запуском).
- Можливо: міграція на PostgreSQL.
